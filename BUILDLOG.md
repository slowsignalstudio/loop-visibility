## 2026-07-08 — Day 5

**Claim graph session (evening).** Started making the dependency graph real, per the sketch in `design/claim-graph-increment.md`. Increment A gives claims identity at write time: **`analyze_recurring`** mints a **`claim_id`** on every price change and **`verify_findings`** carries it through each verdict (null when the caller omitted it, recorded visibly like `"no confidence stated"`). Increment B adds the additive **`0003_claim_edges`** table and **`readClaims()`** in `lib/claims.ts`, the only legal way for one loop to consume another's verified claims: it snapshots the producer's confidence verbatim onto the edge and writes the edge row *before* returning the claims, the same write-before-anything rule as `writeTrace`, with a throw on failure so consumption can never go untraced. The design decision worth remembering: a claim that passed verify but carries no id is not consumable — identity is the precondition for the graph, so the boundary refuses anonymous claims instead of inventing ids after the fact. Nine new tests cover minting, echo-through, extraction, edge ordering, and loud failures; all four checks pass.

Built Level 1 of the fleet supervisor from the design brief: the home page is now a **fleet overview** that rolls every run up to a claim-derived trust signal and answers one question, where do I look. The decision that mattered was the triage rule in **`rollUpRun()`** (`lib/fleet.ts`, pure, nine unit tests): a run is safe only when every doubt raised inside it was *discharged with evidence* — so a verify **reversal** whose hedge named the reversed claim stays safe with a visible badge, while an unverified claim, a hedge verify never resolved, or the sentinel **`no confidence stated`** all route the run to the needs-you group. The old viewer moved intact to **`/run/[runId]`** as the drill-down (Level 3), and clearing the safe majority posts one **`cleared_safe`** review row per run through the additive **`0002_reviews.sql`** table, so the single action still leaves a defensible record per run — job 9 in the brief. A **seed script** (`npm run seed:fleet`) writes 13 synthetic runs through `writeTrace` covering every triage bucket, and all four checks pass. The gotcha worth remembering: a `next build` **bus error** that looked like a code problem was a corrupted platform-native **SWC binary** from an interrupted install — `node -e "require('@next/swc-…')"` isolated it in one line, exactly the debug-from-the-outside-in lesson from Day 3.

Turned the raw JSON viewer into the craft pass. Each hop is now a **`StepRow`**: a one-line summary (phase, tool, plain-English headline, status chip, colored by outcome) that expands to its evidence, backed by a pure **`summarize()`** helper so the messy jsonb-narrowing lives in one testable place. The design core landed in **`VerifyEvidence`**, which shows each verify claim as a card with the raw monthly charges beside the verdict, so the **AWS** reversal reads at a glance: its middle charge ($41.88, neither the claimed old nor new price) is flagged, which is exactly why a clean price step is impossible. Added **Framer Motion** entrance animations and a live "thinking" indicator so the 23-second run reads as progress rather than a frozen screen.

The gotcha that ate an hour and taught the most: rows were written, readable, and arriving in the browser (`poll read 3 rows`) yet the list stayed empty. The cause was a **React Strict Mode** trap — `addRows` deduped by mutating an external `seen` Set *inside* the `setRows` updater, and Strict Mode's double-invocation of updaters added the ids on the first pass then dropped all three as "already seen" on the second. Fix was to make the updater pure (dedupe against previous rows), and the debugging lesson was to surface swallowed errors before guessing. Every step of this is captured in **`LESSONS.md`**.

Day 4 stood up evals and routines. The **eval harness** (`npm run eval`) runs two layers: a **`deterministicScore()`** that runs the tools directly over the committed dataset (instant, free, proves the pipeline), and the **real Sonnet agent** N times scored by **`scoreFindings()`** against ground truth (finds the three real increases, verifies every claim, rejects the AWS trap, total is +$15.50/mo), gated at an 80% pass rate. Extracted the agent loop into **`lib/agent.ts`** so the route and the eval run the same code. Added three **routines** in `routines/`: nightly eval (writes a dated scorecard), daily digest (Haiku), and build-log-to-demo-script (Opus).

## 2026-07-06 — Day 0

Stood up Loop Visibility end to end: scaffolded the app with `create-next-app` (Next 16 / TypeScript / Tailwind / Supabase), pushed it to `slowsignalstudio/loop-visibility`, and deployed the empty shell to Vercel with a streaming Anthropic route (`/api/stream`) proving the model wiring works in production. The decision that actually mattered today was the trace schema — a single `traces` table where every hop stores its **evidence** (`tool_input` / `tool_output`) right next to its **verdict** (`model_confidence` / `verification`), on the principle that the viewer can only ever show what the trace preserved at write time. Created the Supabase project, ran that one table, and confirmed the full loop: inserted a two-hop sample run over the REST API and watched the live viewer render `plan` → `act` → `web_search` from real rows. The one gotcha worth remembering: `@supabase/supabase-js` constructs a realtime WebSocket eagerly and Node 20 has no native `WebSocket`, so the viewer threw until I pinned the project to Node 22 — a small runtime detail that quietly blocked the whole read path. Also set the model policy (Sonnet runs the loop, Haiku for cheap checks, Opus only when quality visibly matters) and marked the column set as an authoritative, sign-off-required design decision, since a column not captured on Day 0 is evidence no future run can recover.

**Build session (evening).** Built the agent loop as one server route (`/api/run`): **Sonnet** calls the three tools (`lookup_transactions` → `analyze_recurring` → `verify_findings`), and every hop writes its trace row *before* the result is fed back, capped by a **10-step guard**. Committed the synthetic dataset — 300 transactions with three planted increases (**+$15.50/mo**) and one trap, **AWS** billed as a usage-metered subscription, so `analyze_recurring` over-counts it and `verify_findings` rejects it — the false-positive correction the demo turns on, and it fired in all three fresh runs (`failed = 1` each). The story of the day: after making `confidence` a **required** field on the act tool (written verbatim to `model_confidence`, or `"no confidence stated"`, never null), the model hedged on the trap *in its own words* — "usage-based cloud billing, not a fixed subscription price" — before `verify` even ran. Shipped the live viewer: it subscribes to `traces` by `run_id` via **Supabase realtime** with a **1-second polling fallback**, rendering each row as raw formatted JSON in a monospace column. Wiped the table and re-ran three times so the trace reads consistently end to end; realtime still needs one `alter publication` line in the SQL editor to flip on, and polling covers it meanwhile.
